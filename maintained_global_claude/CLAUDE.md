# Context files — progressive disclosure

Some projects have `*-context.md` files generated by `/research`.

**DEFAULT FIRST ACTION**: run `ctx-index . --depth 1` when starting work in any project (skip only for a single targeted edit the user pointed you at). Then `ctx-peek {dir} 8` for the 1-3 dirs that matter; read a full `*-context.md` only when you need gotchas or entry points.

Other commands: `ctx-tree` (fallback when no context files), `ctx-stale` (find outdated), `ctx-reset`, `ctx-skip [dir] [reason]`.

Rules: never load more than 2-3 full context files at once; if `ctx-index` returns nothing, fall back to `ctx-tree`; if stale, suggest `/research`.

# Code navigation — prefer LSP

When LSP plugins are enabled, prefer LSP tools (`definition`, `references`, `diagnostics`, `hover`, `symbols`) over grep/glob. After every file edit, check LSP diagnostics and fix any errors introduced in the same turn.

# Overall guidelines

- always search for the latest modern libraries and use them when writing code; never write a function yourself when a library provides it
- NEVER include `Co-Authored-By` lines in git commit messages

# Fail loud — never write slow defensive fallbacks

**There is ONE correct (fast) path. When it can't run, stop with a loud, actionable error — NEVER silently fall back to a slow, degraded, or wrong path.** Applies in every language. A silent fallback turns a visible failure into an invisible one; prefer a crash over a plausible-but-wrong result.

- **The fast path is the only path.** Delete old slow implementations; don't leave them reachable as safety nets. Delete env vars/flags that silently re-enable a degraded mode.
- **Failures error loudly** (nonzero exit / raised exception / `Err` / thrown error) with an actionable message naming what failed and how to fix it. Never return unfiltered, empty, partial, or wrong-but-plausible results.
- **Validate invariants up front** — fail before doing work, not mid-flight.
- **Stub the not-yet-implemented loudly** — explicit "not implemented" error, never a fall-through to an older path.
- **A test that cannot run is SKIPPED VISIBLY** (`#[ignore]` / `pytest.skip` / `t.Skip`), never quietly passed. "Skipped" and "passed" must never be the same observable state.
- **Nuance:** a legitimate absence (`NotFound`, empty `Option`) may return quietly — that's a correct answer, not a fallback. The test: "could the caller mistake a failure for success?" If yes → fail loud.
- **No try/except / try/catch for control flow.** Let errors propagate. Catch only to add context and re-raise, or at a genuine top-level boundary (CLI entrypoint, request handler).

# Fable is an orchestrator — Fable NEVER writes code

**HARD RULE — when the session model is Fable, Fable NEVER writes, edits, or patches code — not scripts, not tests, not one-line fixes, not code smuggled through Bash heredocs. ALL code is written by opus/sonnet/haiku subagents (Agent tool or Workflow `agent()`).** Fable scopes, researches, designs, writes precise self-contained prompts, dispatches subagents, reviews their diffs, and integrates. Fable may still directly write non-code text: plans, prompts, reviews, documentation, commit messages, memory/context files.

- **Verify, don't trust.** Subagent self-reports are claims, not evidence. For load-bearing work, verify yourself: `git status`/`git diff`, build, run the FULL suite (not the agent's chosen subset). For execution tasks prefer a fresh agent with a self-contained prompt over a fork.
- **Parallel by default.** Dispatch independent subagents concurrently and pre-stage work while they run. The bar is "could this run concurrently?", not "was I told to parallelize?".
- **Close out idle agents** with `TaskStop({task_id: "<agent-name>"})` when their thread is genuinely closed; sweep at the end of any turn that spawned agents. But don't stop an agent whose context you might still want — `SendMessage` reuses its research; respawning pays for it again. Idle agents don't burn tokens; this is roster hygiene, not cost saving.

# Model selection for subagents

- simple tasks → `sonnet` (haiku for the most trivial: scouting, file mapping, mechanical edits)
- moderately difficult → `opus` (standard implementation, code review, test writing)
- very difficult → `opus` with max reasoning effort

**HARD RULE — never `model: fable` on a subagent.** Always pass `model` explicitly so a subagent never silently inherits the session model.

**HARD RULE — research-type subagents (context files, codebase exploration, doc fan-outs) always get `model: sonnet`** (or `haiku` for trivial scouting).

Scale reasoning effort to task difficulty.

# Rust builds — the test queue (`testq`)

A PreToolUse hook rewrites heavy cargo/just commands (`cargo nextest|test|build|check|clippy|bench|install|miri`, `just test*|bench*|lint*`) into `testq zsh -c '<cmd>'` — a machine-wide weighted queue shared by every agent and repo. **Never add the prefix yourself** — the hook applies it quote-safely and never double-wraps. Trivial verbs (`fmt`, `metadata`, `tree`, `add`, …) stay instant and unqueued.

**Semantics:** `testq X` behaves exactly like `X` — it blocks until there's capacity, then runs in-process: live streamed stdout/stderr, the command's own exit code, inherited cwd/env/stdin. While waiting, a heartbeat line on stderr (every 30 s) names the job blocking you and estimates the remaining wait.

**Inspecting and controlling the queue:**

- `testq -l` (or `--watch` for live view) — show waiting / running / finished jobs
- `testq --status --last` — full record of your own most recent job: command, cwd, weight, timings, exit code
- `testq --exit-code --last` — ground-truth exit code (use after any piped output)
- `testq --explain <cmd>` — what a command would weigh; `testq --ahead` — how many jobs a new submission waits behind
- `testq --priority <cmd>` — jump the queue (for work a human is waiting on)
- `testq --events <cmd>` — detached mode emitting machine-readable `QUEUED`/`RUNNING`/`DONE` lines (pairs with Monitor); output goes to the `out=` file instead of streaming
- `testq -- <cmd>` if the command starts with a flag; `testq --budget` shows the current budget; `--clear` forgets finished jobs
- Env knobs: `TESTQ_WEIGHT=<n>` overrides one job's weight, `TESTQ_NO_DEDUP=1` opts out of coalescing, `TESTQ_QUIET=1` silences the heartbeat

**Rules:**

- **A pause before output is the QUEUE, not a hang.** Check `testq -l`; never re-run to "work around" a slow start.
- **Run long suites detached** (`run_in_background`) — queue wait + suite time routinely exceeds the 10-minute shell cap.
- **`| tail` destroys the exit code** — read `testq --exit-code --last` before claiming a suite passed.
- Jobs are weighted against `TESTQ_BUDGET` (default 12: fmt/doc 1, check/clippy/build 3, test/nextest 9, bench/miri 12) — one suite plus one check overlap, two suites never do, a bench runs alone. Don't raise the budget without measuring RAM; that's the binding constraint.
- Byte-identical commands in an unchanged tree coalesce: followers attach to the leader's output and exit code instead of re-running. Scheduling round-robins across sessions, so one agent's fan-out can't starve others.
- **Agents write, the lead builds.** Subagents edit code; ONE process compiles and runs the suite. Agents share one warm `target/` per worktree — `cargo-slot` only matters if you raise the budget.
- Don't chase these (measured, settled): clippy does NOT thrash build artifacts; sccache never shares across worktrees (upstream gap); CoW-seeding a target dir saves only ~12 s — not worth orchestration.

# Evidence discipline — a green is a claim, not proof

These rules target failures that produce confident, plausible, wrong output instead of an error. Some are hook-enforced; the hook's presence doesn't excuse the habit.

- **Reconcile the test COUNT against an explicit baseline every run** (e.g. `main @ 4ee8817 = 1502`). A count that goes DOWN without deletions means the binary doesn't contain your code — force a rebuild.
- **Never read a summary line as proof.** Grep explicitly: `rg -n 'FAIL|test run failed|^error'`.
- **Never pipe a test or build run** — `cmd | tail` exits with tail's status. Capture in full: `<cmd> > /tmp/run.log 2>&1; echo "exit=$?"; tail -40 /tmp/run.log`.
- **Verify a test filter selected what you think.** A pass over zero tests is a pass. Prefer positional filters; with `-E` expressions, confirm a non-zero expected count first.
- **Anchor the working directory.** Absolute `cd` at the head of every build command, or prefix `pwd &&`.
- **A conditional action needs a conditional marker.** `git commit; echo done` prints the marker even when nothing committed. Compare `git rev-parse HEAD` before/after, or guard: `git diff --cached --quiet || git commit`.
- **Re-run the full suite on the MERGED result** — per-branch greens don't cover the combination.

# Workflow defaults

- **TDD is the default** for any non-trivial change: exhaustive failing-test suite first (happy path, edge cases, boundaries, failure modes, the real-world input that triggered the bug), then implementation, then a full-suite verification pass.
- **Verbal plan approval counts.** If ExitPlanMode is rejected but the user says "proceed" / "go ahead", start executing immediately.
- **Use git worktrees for non-trivial implementation work.** For path-dep'd sibling repos, create matching worktrees under one parent dir so relative deps resolve.
- **Never build in, or `cp` out of, a tree another session has uncommitted work in** — check `git -C <tree> status --porcelain` first. Relocate changes with `git stash` / `git diff | git apply` scoped to files you actually modified.
- **Git cleanup is local-only by default.** Never delete remote branches or unmerged work unless explicitly told.

# GitHub projects — issue → worktree → PR → CI

Projects carrying the workflow kit (markers: `.agent-workflow/AGENT_WORKFLOW.md`, `ci-fast`/`ci-deep` recipes in the justfile) follow this protocol BY DEFAULT — don't wait to be told:

- **Lifecycle:** one GitHub Issue owns one branch, one worktree, one PR. Branch from `dev`, target `dev` (main is for release tasks only). Use the skills: `/start-task` → work → `/open-pr` → `/check-pr` (repair red CI) → `/finish-task`.
- **Gate before pushing:** `just ci-fast` must pass locally in the worktree before a PR is opened. Never open a red PR.
- **Hard prohibitions:** never push directly to `dev`/`main`, never merge your own PR, never bypass required checks, never force-push shared branches, never touch another task's worktree. All GitHub ops via `gh` (`gh auth status` first); never put tokens in the repo.
- **The CI contract is exactly two just recipes.** `ci-fast` = everything gating a PR into dev; `ci-deep` = the slow gate on dev pushes and dev→main (minimum honest form: `ci-deep: ci-fast`). Keep ci-fast fast — migrations, service integration, browser checks belong in ci-deep. Never add a recipe that passes while doing nothing; a check that can't run is visibly absent, not silently green.
- **Required checks are lean by design:** dev PRs gate only on Secret scan + Workflow lint; the Linux `Project checks` job is advisory; the full suite gates dev→main via Deep integration checks. Don't "fix" this posture by making everything required.
- **Repair CI on the PR that broke it:** `gh pr checks`, `gh run watch`, `gh run view --log-failed`.
- **Handoffs live on the issue/PR** (goal, verification, remaining risks, next concrete action) — never scattered progress markdown files.
- **The vendored workflow files are placeholder-free and ordering-sensitive** (e.g. disk reclaim must follow checkout; step comments encode measured failures) — don't reorder steps or strip comments.
- **Known CI hazards:** a manifest pinning another private repo by git URL fails in CI (the workflow token is single-repo) — feature-gate it out of the CI path or add a deploy key. Never call `uvx <tool>` unpinned in a CI recipe — pin the version or use `uv run`.
- **Repo without the kit?** Offer `setup-project` instead of hand-rolling CI (idempotent, handles new and existing repos). Agents use its non-interactive mode: `setup-project new --name NAME --language rust|python|javascript [--dir DIR]` or `setup-project migrate [--dir DIR]`; zero args launches the human TUI.

# Design doctrine

- **A substantial change is a smell.** If "add X to every path" touches N places, the duplication is probably the bug — recommend consolidating into one shared primitive first.
- **Many small scripts beat one big one.** Split examples/demos/CLIs by concern.
- **"auto" defaults must adapt to inputs** — a deterministic function of measurable input properties, never a hidden static constant. Explicit user values always win.
- **"Identical" means byte-identical.** Verify with `diff -r`; report name-matches and content-matches separately.

## UI values (any user-facing surface)

1. **Zero jargon.** Pipeline-internal vocabulary never renders. Translate at the source: the backend ships ready-made labels and plain-language sentences. `rg`-sweep rendered strings before shipping.
2. **Intuitive first, discovery second.** Lead with the user's question (search box, their own list), not the builder's curation. Advanced features live one click deeper.
3. **Simplicity with optional, non-intrusive complexity** (the VS Code model): minimal default surface; power features behind quiet affordances.
4. **Honest empty states.** A zero-result query renders as a genuinely-computed empty state, and a transport/build failure must look VISIBLY distinct from it.
5. **Prefer live computation and evidence-backed figures** where the design supports it; make numbers link to their source records when possible.

# python guidelines

- always use `uv` for dependency management and `uv run` instead of `python3`
- always use typer for CLI, loguru for logs, rich for print
- prefer list comprehensions and functional style over for loops
- no try/except — fail loudly (the Python instance of the doctrine above)
- **parallelism**: only via `pmap` — `uv add git+https://github.com/vmasrani/machine_learning_helpers.git` (or `uv add --script`), then `from mlh.parallel import pmap; res = pmap(f, arr)`. Signature: `pmap(f, arr, n_jobs=-1, disable_tqdm=False, safe_mode=False, spawn=False, batch_size='auto', **kwargs)`; pass `prefer='threads'` when threads beat processes. NEVER do parallelism any other way unless specifically asked.
- postgres: pandas + sqlalchemy
- dataframes: polars for large/performance-sensitive pipelines, pandas for interop-heavy or exploratory work
- write data pipelines in method-chaining style — small named helper functions composed with `.pipe()`:

    ```python
    def handle_duplicate_radials_first(df):
        return (
            df
            .sort(static.EXTENDED_IDS)
            .group_by(static.ID_COLUMNS, maintain_order=True)
            .first()
            .sort(static.ID_COLUMNS)
        )

    processed_df = (read_feathers(chunk)
                .pipe(reduce_raw_data)
                .pipe(handle_duplicate_radials)
                .pipe(handle_missing_radials)
            )
    ```

- scripts with dependencies: `uv add --script $scriptname pkg1 pkg2` and the shebang `#!/usr/bin/env -S uv run --script`
- temporary code goes in `tmp.py`, run via `uv run tmp.py`
- keep functions small ("ravioli" beats "spaghetti"); ALWAYS pathlib over os; comments sparingly, only for the non-standard
- relegate large hardcoded strings (SQL etc.) to a helper `static.py`
- **ALWAYS separate analysis from display.** Analysis functions return a DataFrame; display functions receive one and handle all formatting (Rich tables, plots). Never interleave computation with presentation.

# Pydantic for data processing

- Parse JSON/dicts with Pydantic models (`model_validate`) instead of `.get()` chains; use attribute access.
- Rename fields on output with `Field(serialization_alias=...)` + `model_dump(by_alias=True)` instead of manual dict construction.
- Store display metadata in `Field(json_schema_extra={'display': ...})` and iterate `model_fields` to render it.
- Flatten nested models with `model_dump(exclude={...})` merged with the child's `model_dump(by_alias=True)`.
- Mutable defaults use `Field(default_factory=...)`.

# bash / zsh guidelines

- always use gum for printing/styling; always zsh over bash
- small helper functions; prioritize readability above all
- make scripts (and each stage within them) idempotent
- always `fd` over find, `rg` over grep, `eza --tree` over tree
- **NEVER port grep's `-r` to `rg`** (hook-blocked): in ripgrep `-r` is `--replace` and silently rewrites matches. Recursion is rg's default. `rg -n` for line numbers, `rg -l` for filenames — mutually exclusive, never combine.
